## 一、项目概述

### 1.1 背景与目标

本系统面向中小型养老机构，以"老人为中心、数据驱动运营"为设计原则，整合入住、财务、人事、药物、仓库、上门服务等核心业务流程，替代现有纸质或分散电子台账，实现全院数字化管理。

核心目标：减少护理漏项、财务误差、跨部门信息断层；对接机构合规报表需求；为管理层提供实时运营看板。

### 1.2 用户角色

角色说明：超级管理员（全权限）。机构负责人可配置系统参数与运营看板；财务人员（缴费、凭证、报账、月结等）；人事专员（护工档案、排班、打卡管理）；护理主管（护工考核、护理计划分配、老人护理记录查看）；护工（打卡、护理记录录入）；仓库员（物资入出库登记、库存盘点）；药剂员（药品管理、发药登记）；上门服务专员（预约受理、服务记录提交）。

## 二、系统架构

### 2.1 架构图（Mermaid）

前端层 → Nginx & 网关层 → 后端服务层 → 基础设施层

- HTTPS/REST + JWT 校验
- Vue3 页面/表单 + Element Plus + Pinia 状态 + Vue Router + Axios REST 请求
- Nginx 反向代理（静态资源+API）
- Spring Cloud Gateway（鉴权/限流/路由）
- Spring Boot API + RBAC 权限校验
  - 入住/老人服务
  - 人事/护工/排班/护理计划
  - 财务服务（缴费/凭证/报账/月结）
  - 仓库服务（入库/出库/盘点）
  - 药物服务（药品/批次/发药）
  - 上门服务（预约/记录/结算）
  - 通知与提醒服务
  - 报表导出服务
  - 操作日志服务（独立落库，保留2年）
- MySQL 主从（业务数据）
- Redis（缓存/会话）
- OSS 对象存储（附件/图片/照片）
- Quartz 定时任务（月结/预警/报表/年龄同步）

### 2.2 整体技术栈

- 前端：Vue 3 + Vite + Element Plus + Pinia + Vue Router + Axios
- 后端：Spring Boot 3.x + Java 17 + MyBatis-Plus + Spring Security + JWT
- 数据库：MySQL 8.0（主从读写分离）+ Redis 7.x（缓存/会话）
- 文件存储：对象存储 OSS（合同附件、凭证图片、老人照片等）
- 定时任务：Quartz（月结算、缴费到期预警、报表生成、年龄同步可选）
- 网关：Spring Cloud Gateway（统一鉴权、限流、路由）
- 部署：Docker Compose / K8s，Nginx 反向代理前端静态资源

### 2.3 前后端分离约定

所有接口遵循 RESTful 规范，统一返回结构：

```json
{
  "code": 200,
  "msg": "success",
  "data": {},
  "timestamp": 1712100000000
}
```

分页统一使用 page、pageSize、total 字段；敏感字段（身份证号、手机号）在传输层脱敏，查看原文需二次鉴权。

### 2.4 权限模型

RBAC（基于角色的访问控制），权限粒度到菜单 + 按钮级别。JWT Token 有效期 8 小时，支持 Refresh Token 无感刷新。操作日志全量记录到独立日志库，保留两年。

## 五、入住管理

### 5.1 入住表单字段（完整）

表单分区：基本信息 / 家属联系人 / 费用信息 / 护理需求 / 既往疾病 / 护理等级 / 主治医疗机构 / 关联护工（新增）

#### A. 基本信息

- 姓名：文本，必填
- 身份证号：文本，必填，18 位校验，唯一性校验
  - 自动解析：性别、出生日期
  - 展示/接口脱敏
- 年龄：数字，由身份证自动计算，可手动修正
  - 标红提示规则（用于界面提示，不限制提交）：
    - 人员类别为"社会化"或"低保对象"
    - 且年龄 < 60（小于 60 周岁）
    - 则年龄字段标红（并展示提示文案）
  - 年龄同步要求：一直入住的老人年龄需要按年份同步更新
- 性别：单选（男/女），可由身份证自动计算，可修正
- 出生日期：日期，由身份证自动填充，可修正
- 籍贯：文本，可选
- 民族：文本，可选
- 照片：图片上传（拍照或上传到 OSS）
- 入住日期：日期，必填，默认今日
- 床位号：下拉，从空闲床位中选择，必填
- 人员类别（新增）：单选（社会化 / 低保对象 / 五保对象），必填
- 是否享受长护险（新增）：单选（是/否），默认否
- 是否使用消费券（新增）：单选（是/否），默认否

#### B. 家属联系人

- 联系人1姓名：文本，必填
- 联系人1关系：下拉（子女/配偶/其他），必填
- 联系人1电话：手机号，必填
- 联系人2姓名：文本，选填
- 联系人2关系：下拉，选填
- 联系人2电话：手机号，选填
- 紧急联系人：勾选联系人1或2，必选一个

#### C. 费用信息

- 合同月费用：金额，必填，精确到分
- 押金金额：金额，选填
- 合同起始日期：日期，默认等于入住日期
- 合同有效期（月数/年数）：下拉
- 合同附件：上传 PDF/图片，存 OSS
- 付款方式：单选（月付/季付/年付/一次性收款）
- 银行账户：下拉，从系统配置账户中选

#### D. 护理需求（多选，可扩展）

压疮护理、大便失禁护理、小便失禁护理、协助洗澡、喂饭、打流食、喂药、康复训练、翻身拍背、导尿管护理、鼻饲管护理、气管切开护理、血糖监测、血压监测、心血监测、临终关怀

#### E. 既往疾病（多选，可扩展）

高血压、脑梗、糖尿病、心脏病、认知障碍症、痴呆、前列腺疾病、癫痫、骨折损伤、压疮、帕金森、慢阻肺、肾功能不全、精神疾病、其他（文本输入）

#### F. 护理等级（单选）

能力完好 / 轻度失能 / 中度失能 / 重度失能 / 完全失能

每个等级对应默认护理计划模板（可在系统管理中配置）。

#### G. 主治医疗机构（选填）

绑定医院、主治医生、常规用药

#### H. 关联护工（新增）

- 护工服务人员（多选）：可选择多名护工用于为老人提供护理服务
- 保存规则：入住提交时写入护工-老人关联（多行记录）

### 5.2 入住流程

1. 填写并提交入住表单
2. 系统校验
   - 身份证唯一性
   - 床位可用性（床位状态需为"空闲"）
   - 关联护工校验（如要求在职/可用）
3. 生成老人档案（唯一编号：YYYYMM + 4位序号）
4. 自动同步费用
   - 向财务管理 - 缴费管理创建一条缴费档案：包含合同月费、入住日期、付款方式
   - 同步记录入住维度标记：人员类别、是否享受长护险、是否使用消费券（用于后续对账/统计）
5. 分配责任护工（新增落地）
   - 读取入住表单"关联护工（多选）"选择结果
   - 创建护工-老人关联记录（多对多）
6. 发送入住通知短信至家属联系人1（可选，需接入短信服务）

### 5.3 老人档案详情页

- 入住基本信息（可编辑）：人员类别、长护险/消费券状态、年龄标红提示状态
- 护理计划（关联护理等级模板）
- 护理记录时间线
- 缴费记录（可显示实收来源拆分：长护险/消费券/其他）
- 用药记录 / 体征记录（血压/血糖/体重趋势图）
- 家属沟通记录
- 附件文档
- 新增展示：当前关联护工列表（主责/备用可选扩展）

### 5.4 退住管理

退住日期、退住原因（正常退住/转院/去世/其他）、押金退还金额计算、费用结清确认、床位状态自动释放

退住联动：建议结束该老人当前在岗护工关联（或标记失效）

### 5.5 转床管理

源床位 → 目标床位，记录转床时间和原因，档案自动更新

## 六、财务管理

### 6.1 缴费管理（核心计费逻辑）

#### 计费规则（应收）

- 规则 A（短期入住，≤7 天）：日费用 = 180 元/天，月应收 = 180 × 实住天数
- 规则 B（正常入住，＞7 天）：月应收 = (合同月费 ÷ 当月自然日数) × 当月实住天数
- 规则 C（月末未住满）：当月多收费用顺延至下月冲抵，不退款（合同约定退款者除外）
- 规则 D（费用预警）：当老人账户余额 ÷ 日均费用 < 7 时，进入首页提醒条；缴费到账后提醒自动解除

#### 月结算流程

每月 1 日 0 点，Quartz 定时任务自动计算上月各老人应收金额并生成月账单草稿。财务人员核对后点击"确认结算"正式生效。支持手动触发重算（当入住信息被修改时）。

#### 缴费登记

手动录入收款记录（现金/转账/POS），支持批量导入（Excel 模板）。每笔收款需上传凭证图片。系统自动更新老人账户余额及预警状态。

新增建议（与长护险/消费券联动）：在缴费登记中支持记录实收来源拆分字段（长护险核销、消费券抵扣、其他收款）。

#### 缴费列表功能

按月份/老人姓名/床位/缴费状态筛选；导出当月对账 Excel；汇总统计（应收/实收/欠收/顺延金额）

### 6.2 凭证管理

收入凭证（缴费）、支出凭证（采购、薪资、水电）分类管理。凭证编号自动生成（年月 + 序号）；支持图片附件；支持按日期/类型/金额区间查询；提供月度/年度凭证汇总导出。

### 6.3 报账管理

员工提交报账申请 → 部门主管审批 → 财务审核 → 打款确认。支持驳回并填写原因；报账状态追踪（待审批/审批中/已完成/已驳回）；与凭证管理联动，审核通过自动生成支出凭证。

## 七、人事管理

### 7.1 护工管理（新增联动规则）

护工档案字段：姓名、身份证、性别、年龄、学历、联系电话、紧急联系人、入职日期、岗位类型（全职/兼职）、护理资质证书（图片上传）、工资结构（基本工资+绩效）、分管老人列表。

功能：档案 CRUD、离职办理（记录离职日期和原因，自动解除分管关系）、护工-老人分配管理（多对多）、护工绩效评分录入（每月由护理主管填写）。

新增关键交互要求（基于入住管理的选择结果）：

1. 在 入住管理 的表单中可选择多名护工为老人服务并提交
2. 系统写入护工-老人关联记录
3. 在 护工管理 二级菜单的列表/详情中展示该护工护理的老人数量（统计该护工当前关联的在住老人）

补充规则（建议实现）：
- 统计口径：仅统计老人状态为"在住/护理服务中"的关联数量
- 退住或护工离职联动：结束或失效关联记录，使统计数量自动减少

### 7.2 打卡记录

支持接入第三方考勤设备（API 对接）或手动录入；展示每日打卡明细（上班/下班）；统计月度出勤天数、迟到/早退/缺勤次数；导出月度考勤报表；与薪资模块联动（超出部分自动扣款，可配置规则）。

### 7.3 排班管理（补充功能）

早班/中班/夜班三班制；支持拖拽式排班日历；排班冲突检测（同一护工同一时段重复排班）；节假日自动标注；排班表导出打印。

## 九、药物管理

### 9.1 药品档案

药品名称、通用名、规格、剂型、厂家、批准文号、有效期、存储条件、是否处方药

### 9.2 药品入库

批次管理（同一药品多批次入库）、批号、有效期录入；近效期预警（默认提前 30 天，可配置）

### 9.3 发药管理

按老人开具发药单；支持定时发药提醒（关联护理计划中的喂药需求）；发药记录（药品名/剂量/时间/执行护工）；历史查询按老人/药品/时间段筛选

### 9.4 用药档案

每位老人的常规用药列表（由医护人员维护）；药物过敏记录；药物相互作用提示（录入多种药物时系统提示常见配伍禁忌）

### 9.5 统计报表

月度药品消耗量、按老人消耗汇总、近效期/过期药品清单

## 十一、系统管理

### 11.1 权限管理

- **菜单管理**：树形结构，支持添加/编辑/删除菜单节点，配置菜单图标、路由、排序
- **角色管理**：角色 CRUD，为角色分配菜单权限和按钮权限
- **按钮权限**：精确到"新增/编辑/删除/导出"等按钮级控制

### 11.2 账户管理

账户 CRUD、重置密码、绑定角色（支持多角色）、账户启用/停用、最后登录时间/IP 展示

### 11.3 系统配置（超级管理员可见）

- 机构基本信息：名称、地址、联系方式、营业执照
- 计费规则：短期日费率（默认 180 元）、预警天数（默认 7 天）
- 护理等级模板：各等级默认护理计划项
- 床位管理：楼栋/房间/床位号维护
- 供应商管理：采购常用供应商档案
- 短信配置：接入短信服务商 API Key
- 数据字典：民族、疾病、物资分类等字典值维护
- 年龄标红阈值：ageWarningThreshold 默认 60（用于 UI 标红提示，不作为硬校验）
- 长护险/消费券策略开关（可选）：enableLongCare、enableCoupon 默认开启（用于前端展示与统计维度）

### 11.4 操作日志

## 十四、非功能性需求

- **性能**：列表页接口响应 ≤ 500ms（P95），支持 100 并发用户
- **安全**：XSS / SQL 注入防护；敏感字段加密存储（身份证、手机号 AES 加密）；密码 BCrypt 哈希；HTTPS 强制
- **兼容性**：前端支持 Chrome / Edge / Firefox 最新两个大版本；分辨率适配 1366×768 及以上
- **可用性**：年可用性 ≥ 99.5%；支持数据库自动备份（每日增量，每周全量）
- **扩展性**：业务模块通过 Spring 事件机制解耦（如入住成功触发费用同步），方便对接第三方系统（医疗 HIS、民政平台、家属 App）

### 16.1 核心关联：护工-老人

#### 表：t_elderly（老人）

| 字段 | 说明 |
|---|---|
| id(PK) | 主键 |
| unique_no | 唯一编号 |
| name | 姓名 |
| id_card_enc | 身份证（加密） |
| birth_date | 出生日期 |
| gender | 性别 |
| region/ethnicity | 籍贯/民族 |

| bed_id | 床位ID |
| status | 状态（在住/退住/转床中等） |
| category | 人员类别（SOCIAL/LOW_BAO/WU_BAO） |
| enable_long_care | 是否享受长护险 |
| enable_coupon | 是否使用消费券 |

#### 表：t_staff（护工）

| 字段 | 说明 |
|---|---|

| name | 姓名 |
| id_card_enc | 身份证（加密） |
| gender | 性别 |
| birth_date/age | 出生日期/年龄 |
| phone_enc | 手机号（加密） |
| hire_date | 入职日期 |
| job_type | 岗位类型（full/part） |
| qualification_urls | 资质证书 |
| status | 状态（在职/离职/可用等） |

#### 表：t_staff_assignment（护工-老人关联）

| 字段 | 说明 |
|---|---|
| id(PK) | 主键 |
| elderly_id(FK) | 老人ID |
| staff_id(FK) | 护工ID |
| assign_type | 关联类型（PRIMARY/SECONDARY） |
| start_time | 开始时间 |
| end_time | 结束时间 |
| status | 状态（ACTIVE/INACTIVE） |

约束建议：elderly_id + staff_id + status=ACTIVE 唯一（防止重复关联）

#### 关键落库行为

- 入住提交保存多名护工：写入多行 t_staff_assignment
- 退住或护工离职：对关联记录置为结束/失效，使护工统计数量自动变化

#### 关键接口（建议）

- `POST /elderly`：入住创建（请求体包含 staffIds: number[]）
- `PUT /elderly/{elderlyId}/assign-staff`：入住后更新护工（可选）
- `GET /staff/{staffId}/elderly-count`：护工负责老人数量
